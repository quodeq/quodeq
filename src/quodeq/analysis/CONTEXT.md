# Analysis Context

This context defines the terminology used within the evaluation and orchestration layers of Quodeq.

## Language

**Analysis**:
The orchestration engine that executes the evaluation pipeline, managing AI subagents, prompts, and the execution flow.
_Avoid_: The core.

**Subagent**:
An autonomous or semi-autonomous AI agent specialized in a specific task within the Analysis pipeline (e.g., identifying evidence, performing a security scan).

**Pipeline**:
The sequence of execution steps (e.g., discovery -> analysis -> projection) that constitute a complete evaluation run.

**Prompt**:
The structured instruction set provided to LLMs to drive subagent behavior.

**Thought Chain**:
The reasoning process or trace generated by an AI subagent during its execution.

## Relationships

- The **Analysis** engine orchestrates **Subagents** using specific **Prompts**.
- A **Pipeline** is the realization of an **Analysis** run.

## Flagged ambiguities

- "Analysis" was previously used as a generic term for the entire process — resolved: **Analysis** now specifically refers to the orchestration layer.

## Architecture: Incremental Evaluation Consistency

### Core promise

**Same fingerprint → same finding, always.** A fingerprint is `(file content, dimension, standards, prompts, model)`. If a re-run sees the same fingerprint for a file, it must return the file's existing findings without re-analysis. Incremental == fresh, by construction.

### Storage model

The system has three layers. Only one stores findings; the others describe what happened in a run.

```
  ~/.quodeq/cache/results/{fingerprint}/entry.json     ← SINGLE source of truth
    - findings list (possibly empty — "empty" is a real answer)
    - written once, when a worker emits file_done="ok"
    - durable across runs
    - synchronous + atomic write; never lost on cancel

  <run>/manifest.jsonl                                  ← per-run snapshot
    - {file, fingerprint, status} per source file
    - status ∈ {cached, done, error, pending}
    - workers append as they finish

  <run>/status.json                                     ← per-run verdict
    - state, exit_reason ∈ {done, deadline, cancelled, crashed}
```

Everything else (per-dim Evidence, score, coverage %, dashboard grades) is **derived** from these three at read time. Derived artifacts may be memoised to disk but they are never the source of truth.

### Run lifecycle

```
  PER-FILE   → cache write on file_done="ok"
                (write-once; survives the run)

  PER-RUN    → manifest enumerates every source file
                + status.json records HOW the run ended

  NEXT RUN   → recompute fingerprints; cache hits replay;
                misses get dispatched; same fingerprint → same finding
```

### Derived quantities

```
  Evidence(dim) = ⋃ cache[m.fingerprint].findings
                    for m in manifest where m.status ∈ {cached, done}

  files_read    = | { m in manifest where m.status ∈ {cached, done} } |
  source_files  = | manifest |
  coverage      = files_read / source_files
  score         = compute_score(Evidence, source_files)
  is_partial    = (exit_reason != "done") or (files_read < source_files)
```

`compute_score` thresholds against `source_files` (the total, not the analyzed count). A partial run with thin findings against full-project thresholds naturally falls to "Insufficient". No special-case logic.

### Invariants

The architecture stands on six promises. If any of them break, incremental consistency breaks.

- **I1.** Same fingerprint → same finding. (Cache is content-addressed.)
- **I2.** A file with `file_done="ok"` ends up in the cache before the run ends. (Synchronous, atomic write.)
- **I3.** Every source file under a dim appears in that run's `manifest.jsonl`.
- **I4.** Scoring uses `source_files` for confidence thresholds. Partial runs hit "Insufficient" by formula, not by special case.
- **I5.** `exit_reason` is set on every run end before lifecycle finalises.
- **I6.** The dashboard renders the partial state visibly when `is_partial` is true. Score is still shown — I4 makes it honest.

### What this architecture removes

| Anti-pattern | Why it's gone |
|---|---|
| `<run>/evidence/{dim}_evidence.jsonl` as findings storage | Derived from cache + manifest. Cannot drift from cache because it is computed from cache. |
| `<run>/events.jsonl` as a parallel findings store | Same. SQL projector reads cache + manifest directly. |
| `FindingsRouter` ↔ `_emit_cached_findings` symmetry burden | One write path: worker → cache. The cache write IS the event. |
| `deduplicate_jsonl` post-pass | Duplicates impossible — cache is keyed by fingerprint. |
| Three scoring callsites kept in sync manually | All derive Evidence from the same cache. Symmetry by construction. |
| `_marker: file_done` lines as a side-channel | The cache write IS the marker. No third state. |

### Status (2026-05-21)

The architecture above is the target. Phase 1 (cache write atomicity, honest `files_read`, `exit_reason` on every run end, dashboard Partial badge) is the first compatible step; the legacy JSONL / events.jsonl stores still exist and continue to work, but the cache becomes the authoritative writer for completed files and the source for `files_read`. Phase 2 collapses the legacy stores into derived views.
